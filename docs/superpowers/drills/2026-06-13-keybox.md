# Keybox CLI — Drill Report

Date: 2026-06-13 | Status: **GO**

## 1. Problem Restatement

A developer needs to manage multiple types of credentials (LDAP passwords,
internal system tokens, OAuth client secrets, LLM provider API keys) across two
environments: a Windows desktop and an Ubuntu machine accessed via SSH (headless,
no GUI). Today, credentials are scattered across plaintext files, environment
variables, and memory — insecure and inconsistent.

Drill questions and user responses:

| # | Question | Response |
|---|---|---|
| 1 | Typical usage scenario: local + SSH sync, or independent per-machine? | At minimum independent config; sync is nice-to-have |
| 2 | Unlock mechanism preference? | Three-tier: system-bound (DPAPI/machine-id), password, file-hash |
| 3 | Three tiers: three independent stores or one store with mixed levels? | Independent stores — prevents cross-level compromise |
| 4 | Storage path convention vs config profiles? | Convention: `~/.config/keybox/` with `secret/`, `confidential/`, `top-secret/` subdirs |
| 5 | CLI interaction style? | `keybox [--level] <add\|get\|list\|delete> <domain> <account>` |
| 6 | How to specify security level? | `--secret` / `--confidential` / `--top-secret` flags, mutually exclusive, default `--secret` |
| 7 | Default level if none specified? | `--secret` |
| 8 | `get` output mechanism? | stdout by default, `--clipboard` and `--env` flags |
| 9 | Init flow for security tiers? | Auto-init for secret; explicit `init` for confidential and top-secret |
| 10 | Daemon for key caching? | Secret: stateless; Confidential and Top-Secret: separate daemons |

## 2. Context & Current State

- **User:** Individual developer, daily switching between Windows desktop and Ubuntu SSH
- **Scenarios:**
  1. Manually looking up credentials on either platform
  2. Coding agent on Ubuntu SSH needs to inject credentials into bug-management system
- **Current approach:** Credentials scattered across `.env` files, shell variables,
  plaintext files. No unified tool.
- **Pain frequency:** Daily — every internal system login, every new token setup.

## 3. External Landscape

| Tool | Cross-Platform | Headless Linux | Automation | Gap |
|---|---|---|---|---|
| Bitwarden CLI (bw) | Yes | Partial (session keys) | Yes (API key) | Requires account; two-step login+unlock |
| 1Password CLI (op) | Yes | Service Account req'd | Yes (op run) | Paid; service accounts are paid-tier |
| pass / gopass | Yes (Linux best) | Yes | Yes | GPG dependency; weak Windows support |
| Python keyring lib | Yes | **No** (D-Bus req'd) | No | Fails on headless Linux; not a CLI tool |
| HashiCorp Vault | Yes | Yes | Yes | Massive overhead; requires running server |
| SOPS | Yes | Yes | Yes | Designed for config file encryption, not credential lookup |
| KeePassXC CLI | Yes | Yes | Partial | File-lock conflicts in scripts |
| age/rage (building blocks) | Yes | Yes | Yes | Not a credential manager; excellent crypto foundation |

## 4. Gap & Value Analysis

**Core gap:** No tool exists that is a single binary, works identically
cross-platform (including headless Linux), supports both interactive and
automated use, and requires zero external infrastructure (no accounts, no
servers, no GPG key ceremony).

**Value proposition:**
- Developer personal credential management, local-first, no cloud dependency
- CI/CD and coding agent secret injection via process-env pattern
- Seamless Windows ↔ Linux portability

**Competitive risk:** gopass improving Windows UX or 1Password offering free
service accounts could reduce differentiation. Neither appears imminent.

## 5. Balanced Assessment

| Strengths | Weaknesses / Risks |
|---|---|
| Real, verified need — user hits this daily | Market is niche (individual developers) |
| Clear tech stack: Rust + age, both mature | age Rust crate maturity needs verification |
| Three-tier security model is novel | Three tiers may be over-engineering for most users |
| Single binary, zero external deps, trivial deploy | Linux machine-id binding is not hardware-grade security |
| Daemon model (like ssh-agent) is familiar | Daemon adds complexity and bug surface |
| gopass proves the file-in-Git model works | Sync is v2 scope; v1 is local-only |

## 6. Recommendation: GO

**Rationale:**
1. Gap is real and confirmed by competitive analysis — no existing tool meets all criteria
2. Technically feasible — Rust + age provides a solid foundation
3. User has concrete, daily usage scenarios
4. Scope recommendation: v1 focus on secret + confidential tiers; top-secret can follow. Linux machine-id binding is acceptable for v1; TPM support is a future enhancement.
